Background

What is RADAR?

RADAR — Remote Analysis and Diagnosis tool — is the central platform Philips service engineers use to monitor, diagnose, and resolve issues in medical systems installed across hospitals worldwide.

10+ Years of legacy tool, unchanged
3 Medical modalities (IxR, MR, CT)
3 Continents of user research
4 Distinct user archetypes
Legacy RADAR (2014 R4) alongside the redesigned RADAR 2.0 monitoring dashboard

Business Objectives

  • ↑ Increase number of issues resolved by lowering resolution time per case
  • ↓ Lower overhead costs — reduce frequency of costly field visits
  • → Gain confidence to migrate users from a deeply familiar legacy workflow

Core Challenge

Users had been using the legacy RADAR tool unchanged for over a decade. Any redesign had to respect their deeply embedded mental models while substantially improving efficiency and expanding capability. Change resistance was real and had to be designed around, not against.

Methodology

Design Process

An Agile, user-centered process structured around five phases — with iterative loops between design and testing.

Design process — Background & Research → Definition → Design Exploration → Test & Iterations → Launch & Next Steps (Agile)

Synthesized all research into an affinity map, clustering findings into three guiding design principles. Created information architecture maps per modality and designed optimized navigation models for two primary workflow scenarios.

3

Moved from hand-sketched layout wireframes to high-fidelity visual mockups. Prototyped key interactions for user testing, with special focus on the monitoring dashboard, system details page, and the novel visual LOG timeline.

4

Test & Iterations

User Testing · Iterations · Design Specifications

Periodic feedback sessions with in-house and remote users during each incremental release. Minimal-intervention observation to assess discoverability. Created CSS-ready design specifications for the development team — layout grids, typography scales, and color tokens.

5

Launch & Next Steps

Incremental Roll Out Plan · Development Support

Designed a 7-phase incremental rollout to keep the legacy tool running in parallel during the transition — reducing user risk and allowing gradual adoption. Provided ongoing development support post-launch.

Research Findings

After extensive interviews across IxR (Americas–Europe–India), MR (China–Europe–India), and CT (Europe–India), findings were classified into two core workflow stages.

Pre-Analysis Locate
  • Login process is too complex — dual-step authentication (AOK + RADAR)
  • No need to see all countries and regions — users work in pre-defined territories
  • Cannot customise columns in the system table
  • Direct search by Serial Reference Number (SRN) is the preferred entry point
  • System reachability must be the primary status indicator
Analysis Analyse
  • Too many irrelevant error messages in the LOG file — noise overwhelms signal
  • Users cross-correlate LOG parameters to evaluate system health
  • Status overview is highly valued at the start of a workflow
  • Users need to understand system history across restarts and events
  • Existing LOG viewer is old — engineers use external tools (CAT, UltraEdit)
Research findings — clustered into Locate and Analyse stages with supporting user quotes

Definition Phase

From Insights to Principles

All research findings were clustered into an affinity map — revealing three foundational design principles that guided every decision.

Affinity map — clustering tele-interview and shadowing findings into three design principles
UX Principles — how the three principles interconnect and reinforce each other
Structured Information

Information should be easy to locate. Primary information upfront and given higher priority. Secondary information accessible but discreet.

  • Modules and tools should be easily locatable (1.1)
  • Interface should convey progress of the user's workflow activity (1.2)
  • Information should be prioritised — non-relevant info on second level (1.3)
  • Information layout should augment the user's mental model (1.4)
Adaptable Interface

Interface-level customisations should be possible for the user. Each user has their own work style — the design should adapt to it.

  • Interface designed around the typical workflow of a user (2.1)
  • Key activities treated as milestones to locate oneself in the workflow (2.2)
  • Allow customisations so every user feels comfortable (2.3)
  • Abstract information based on logged-in user context (2.4)
Optimized Navigation

Navigation should be optimised so the novice user feels guided and the expert user feels in control at every step.

  • User should clearly understand where to proceed next (3.1)
  • Visual cues present for quick identification of actions (3.2)
  • Navigation model should correlate with the typical user workflow (3.3)
  • Keyboard and interaction shortcuts to accelerate expert task completion (3.4)

Key Design Elements

Specific innovations in the design that directly addressed the research-discovered pain points.

Global Stats panel + monthly health calendar with three-state colour coding (healthy / warning / FSE onsite)
Visual LOG timeline — activities and alerts charted over time with an interactive cursor, inspired by disassembler/debugger tools (OllyDBG)

The Visual LOG Timeline

The single most impactful design innovation. Engineers previously opened downloaded LOG files in text editors like UltraEdit and manually scanned thousands of lines for errors — a slow, error-prone, cognitive overload exercise.

I proposed a visual timeline that charts activities and alerts over time, with an interactive cursor that lets users jump directly to the line of interest. Activities in LOG files occur as paired events (System Boot → System Shutdown) — I used the visual metaphor of tied brackets familiar from disassemblers like OllyDBG, which the engineers already knew.

This directly reduced analysis time by eliminating the need to scroll through thousands of log lines manually — users could see at a glance where in time an issue occurred, and jump there with a single click.

System Modes

Research Translation

Research findings were translated into three distinct operational modes — Auto, Manual, and Predictive — designed to serve both novice and expert users.

Three system modes — Auto (guided wizard), Manual (expert workflow), Predictive (proactive alerting)
Auto Mode

Guided workflow with a wizard-like interaction. Makes routine tasks easier for novice engineers — Dashboard → System & Issue Details → Resolve.

Manual Mode

User-selectable modules and workflow. System and performance graphs. Enables specialised workflows for expert engineers — Dashboard, Extended System Details, LOG Viewer, Create Case.

Predictive Mode

Intelligent dashboard with proactive and advance alerting. Minimises system downtime and increases service efficiency by anticipating failures before they occur.

The RADAR 2.0 redesign replaced a decade-old tool with a system that met engineers where they worked — across three continents, three modalities, and four distinct user types — without disrupting the deep mental models they had built over years. The incremental rollout strategy proved critical: users who were initially resistant to change became advocates as each phase delivered clear, tangible improvements to their daily workflow. The tool remains in active use with ongoing iterations.
