Case Study: RADAR Dashboard

Remote Analysis & Diagnosis Dashboard for Global Teams

How I built a comprehensive monitoring and alert management system for 1,600+ medical devices installed worldwide, enabling service technicians to respond faster to critical failures.

1,600+ Radiology Systems

The Challenge

Fragmented Visibility

Service engineers had to log into multiple disconnected systems to understand the health of installed systems. Critical alerts were buried among non-urgent notifications.

Slow Diagnostics

Without real-time data from devices, technicians had to make phone calls or wait for manual reports. This delayed problem identification and increased downtime.

Limited Actionability

The dashboard showed problems but didn't guide engineers toward solutions. Developers expected deep technical logs; service leads wanted high-level status summaries.

Global Scale Issues

Systems deployed across multiple regions had different time zones, languages, and connectivity constraints. One-size-fits-all design wouldn't work.

Research Insights

  • Service technicians spent 40% of their time searching for relevant device data across systems
  • Critical failures took 90 minutes average to diagnose (goal: 30 minutes)
  • Field engineers relied on phone support for interpretation, creating bottlenecks
  • Engineers needed both high-level overview AND deep technical detail without clutter

The Solution

A tiered information architecture that balances overview visibility with deep diagnostic tools.

1

Smart Alert Prioritization

Implemented machine-learning driven alert categorization that automatically separates critical failures from routine maintenance alerts.

  • Critical alerts highlighted with context-specific actions
  • Grouped related alerts to reduce alert fatigue
  • Predictive alerts flagged issues before they escalated
2

Multi-Level Information Design

Created a progressive disclosure system that showed summary data by default but allowed engineers to drill into technical logs with one click.

  • Service lead dashboard: High-level device status + team workload
  • Technician dashboard: Prioritized alerts + quick actions
  • Debug mode: Full technical logs + system telemetry for developers
3

Contextual Device Timeline

Designed an interactive timeline showing device event history, helping engineers understand problem patterns and root causes.

  • Visual event log with severity indicators
  • Ability to correlate events across device components
  • Export timeline data for forensic analysis
4

Localized & Accessible

Built multi-language support and ensured the dashboard worked on low-bandwidth connections common in remote service locations.

  • Automatic language detection based on user profile
  • Progressive data loading for slow networks
  • Dark/light mode + high contrast accessibility options
5

Collaborative Workflows

Enabled teams to collaborate on complex diagnostics by sharing alerts, annotations, and recommended actions with colleagues.

  • Inline comments on alerts with @ mentions
  • Knowledge base integration for documented troubleshooting steps
  • Escalation workflows for cases requiring management approval
6

Proactive Notifications

Moved from reactive dashboard checking to proactive multi-channel alerts via email, SMS, and in-app notifications.

  • Customizable notification rules based on device type & severity
  • On-call scheduling integration for automatic escalation
  • Alert fatigue reduction through intelligent grouping

Design Process

Phase 1: Discovery (Month 1-3)

Research & Validation

Conducted 15+ interviews with service technicians, engineers, and service managers across 5 countries. Analyzed 6 months of support tickets to understand real pain points. Built personas based on user archetypes and behaviors.

Phase 2: Exploration (Month 4-6)

Wireframing & Prototyping

Created 5 different dashboard layouts exploring different information hierarchies. Tested prototypes with power users in low-fidelity and interactive formats. Validated alert prioritization logic with machine learning team.

Phase 3: Iteration (Month 7-12)

High-Fidelity Design & Testing

Built comprehensive design system for dashboard components. Conducted 3 rounds of usability testing with real users. Refined visual hierarchy, color coding, and interaction patterns based on feedback.

Phase 4: Implementation (Month 13-18)

Handoff & Validation

Worked with engineering team to implement design specs. Conducted QA testing across browsers and devices. Gathered feedback during soft launch and iterated before full rollout.

Ongoing: Monitoring & Optimization

Post-Launch Learning

Tracked adoption metrics and user behavior. Conducted follow-up interviews to understand real-world impact. Implemented refinements based on actual usage patterns.

Key Learnings

Listen to Power Users

Service technicians had deep domain knowledge. Their feedback on alert classification and prioritization logic was invaluable and led to the ML partnership.

Progressive Disclosure Scales

Showing the right data at the right time meant everyone—from service leads to developers—could use the same platform effectively.

Global Scale Changes Everything

Time zones, languages, and connectivity constraints were not afterthoughts. They shaped every design decision from the start.

Measure What Matters

Adoption rates and satisfaction scores were important, but reducing diagnosis time by 30% was the metric that actually proved business value.
