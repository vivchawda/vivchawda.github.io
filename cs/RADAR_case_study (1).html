Executive Summary

The Challenge: Breaking Legacy Resistance

RADAR was a critical diagnostic platform used for MRI, CT, and X-Ray systems. It had remained unchanged for 10 years, causing massive friction and operational costs ($800–$1,500 per field visit).

🎯

Success Metrics

  • Target resolution time: 29min
  • Field visit frequency: -25%
  • NPS Target: >70 (Baseline: 32)
Problem Definition

The Complexity Ecosystem

User Frustrations

Engineers navigated 8–12 screens per workflow. Critical data was buried, forcing constant context switching.

Behavioral Barriers

Senior engineers (45+) were deeply resistant to changing a tool they had "memorized" for a decade.

"I've been using RADAR for 8 years. I know exactly where everything is. Every time they 'improve' something, it just slows me down."
Discovery Phase

Deep User Immersion

1. Stakeholder Interviews

12 structured interviews with Portfolio Managers to understand strategic objectives and cost drivers.

2. Contextual Inquiry

32 in-depth shadowing sessions across Americas, Europe, and Asia Pacific watching engineers diagnose real issues.

3. Legacy System Audit

Screen-by-screen documentation of MR, CV, and CT modality variants to preserve high-value technical legacy.

Insights

3 Core Theme Clusters

🏗️ Information Structure

Critical info was 4–5 clicks deep. Hierarchy mismatched user priorities.

🎛️ Adaptability

No personalization based on region or role. Settings reset on every login.

🧭 Navigation

Tree navigation was ignored for search, but search lacked advanced filters.

Strategic Design

From UI Refresh to Adaptive System

The Reframe

The problem wasn't "ugly UI"—it was inflexibility. Users created workarounds because the tool didn't adapt to their specific modality or case type.

✅ Prioritized

Workflow efficiency, familiar patterns, and integrated LOG viewing.

❌ Deprioritized

Radical visual redesign (high fatigue risk) and mobile responsiveness.

IA & Execution

Workflow Innovation

Unified Navigation

Consolidated 3 modality-specific structures into one flexible framework with persistent context panels.

The LOG Viewer

Borrowed the visual metaphor of debugging tools (OllyDBG) to create a timeline visualization for log analysis.

"I can analyze the LOG file without leaving the platform? This changes everything." — Usability Participant
